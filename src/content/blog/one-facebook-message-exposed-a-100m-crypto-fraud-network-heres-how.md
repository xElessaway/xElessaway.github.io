<p>It started with a Facebook message from someone I’d never met.</p><p>“Helen Moran”, no mutual friends, no profile picture worth trusting, no reason to exist in my inbox. The message was short and felt like an accident, the kind of thing you might forward to the wrong person:</p><blockquote>USDT Balance: 3,804,650.74<em><br></em>Lucas, your account balance has been deposited.<em><br></em>Account: qqq226 / Password: sss247<em><br></em>Login: <a href="https://btczio.com/">https://btczio.com</a><em><br></em>Please keep this safe.</blockquote><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*fly5lN-kyZrW9YFC1IorEg.png" /></figure><p>I wasn’t Lucas. She wasn’t talking to me. And that balance, three point eight million dollars in USDT, was never real.</p><p>But the domain was. And that’s where this gets interesting.</p><h3>Step 1, I Just Googled the Domain. Then It Got Weird.</h3><p>My first instinct was the same as anyone’s: what even <em>is</em> <a href="http://btczio.com/">btczio.com</a>? I didn’t click the link, <strong>never do that</strong>, but I did want to know what I was looking at before I touched anything.</p><p>First stop was <a href="https://www.virustotal.com/gui/domain/btczio.com"><strong>VirusTotal</strong></a>. Before doing anything else, you check reputation. Drop the domain in, see if any of the 90+ security vendors have already flagged it.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*SekTvxo-DSXO4j38RFhplw.png" /></figure><p>Zero detections. Clean across every engine. That’s not reassuring, it means this domain is fresh enough that the security industry hasn’t caught up yet. New infrastructure is the most dangerous kind, because all the automated defences are blind to it. The absence of flags was itself a flag.</p><p>So I dropped it into <a href="https://urlscan.io/search/#btczio.com"><strong>URLscan.io</strong></a>, a free tool that visits websites on your behalf inside a sandbox and takes screenshots, records all the HTTP requests, and shows you exactly what the page does without you ever loading it yourself.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*EwlSpGVHs44tQ18nKcn8xw.png" /></figure><p>The page looked like a cryptocurrency exchange. Clean mobile-first interface, price tickers, a login/register button. Slick enough that if you received those credentials in a message from someone you trusted, you might actually try them.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*MVDhxK-CJDr2oDAw7-DR0g.png" /></figure><p>I also threw it into <a href="https://app.any.run/tasks/a3eda678-ca65-41d0-9269-0924b1e4b27f"><strong>Any.run</strong></a>, an interactive malware sandbox that lets you browse the site in a real browser environment and watch every network connection it makes in real time.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*Q2lHJbIP_7j6mmiaceDJiw.png" /></figure><p>That’s where the first clue fell out. The Any.run network capture showed the site making API calls to a backend. A contact form or support popup contained an email address embedded in the page source:</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*nNj7z5jVGQoa-NRjU-e_BQ.png" /></figure><pre>ethtrx033@gmail.com</pre><p>An operator contact. The scammer’s email. I now had three seeds: a domain, a Gmail address, and a set of fake credentials.</p><p>Time to find out who was actually running this thing.</p><h3>Step 2, The Domain Told Me Everything Before I Even Asked</h3><p>The fastest thing you can do with an unfamiliar domain is look up its DNS records. Not just the IP address, everything. The nameservers, the mail configuration, the SOA record, all of it. Every one of those fields is a choice the operator made, and choices leave fingerprints.</p><pre>dig btczio.com A +short</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*nXqJTTNUwfv-XV2LkLs5IQ.png" /></figure><p>The IP came back as 35.197.129.159. That&#39;s Google Cloud Platform, GCP, us-east1. Fine, lots of legitimate sites use GCP. But then I noticed the TTL.</p><p><strong>One second.</strong></p><p>Not sixty seconds. Not three hundred. <em>One.</em> A normal website uses a TTL of 300 to 3600 seconds. One second means the domain operator is changing the IP address faster than any blocklist on Earth can keep up. By the time a security vendor updates their database, the record has already rotated. This technique has a name: <strong>fast-flux DNS</strong>, and it’s almost exclusively used by people who don’t want to be found.</p><p>This was no longer just a fake website. This was deliberate evasion infrastructure.</p><pre>dig btczio.com NS +short</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*bjeUVgMxzrTFxXdaq9x1TQ.png" /></figure><p>The nameservers were ns1.julydns.com and ns2.julydns.com. I&#39;d never heard of <a href="http://julydns.com/">julydns.com</a>. Most people haven&#39;t. It&#39;s not a real DNS hosting company, it&#39;s a purpose-built piece of infrastructure that exists specifically to manage DNS zones for scam domains. I had just found the first layer of the network below the surface.</p><h3>Step 3, Pull One Thread, Find a Wardrobe</h3><p>I looked up julydns.com the same way I&#39;d looked up btczio.com. DNS records are public. Anyone can read them. The question is just whether you know what you&#39;re looking for.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*yzdWwcyKjk1uNrx2g9SQ-Q.png" /></figure><p>Three things jumped out immediately.</p><p>First, the IP hosting julydns.com, 47.76.214.40, was <strong>Alibaba Cloud</strong>. Not GCP anymore. Aliyun. China&#39;s dominant cloud provider. The scam domain lived on American servers. The infrastructure running the domain was in China.</p><p>Second, the <strong>MX record</strong>, the mail server for <a href="http://julydns.com/">julydns.com</a>, pointed to mail.chaicp.com. An MX record tells you where email for a domain gets delivered. The fact that julydns.com had one at all was interesting; DNS providers don&#39;t usually send email. The fact that it pointed to an entirely different domain, chaicp.com, meant that domain was also part of this operation.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*brTT90FRa_vQPVXH6HvUKw.png" /></figure><p>Third, the <strong>SPF record</strong> listed two IP addresses for authorized mail sending: 120.55.40.103 and 47.242.196.125. Both Alibaba Cloud. Both China-registered. The entire email infrastructure was Chinese.</p><p>But then I looked at something most people skip.</p><pre>dig btczio.com SOA +short</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*LB4gYGSmIXwbazpN20Ie5g.png" /></figure><p>The <strong>SOA hostmaster</strong> field, an administrative contact embedded in every DNS zone, read master.jmdns.com. Not master.julydns.com. A completely different domain: <a href="http://jmdns.com/"><strong>jmdns.com</strong></a>. This was the management layer sitting behind <a href="http://julydns.com/">julydns.com</a>. The control panel for the control panel.</p><p>I kept pulling.</p><h3>Step 4, The Backend That Runs Everything</h3><p>chaicp.com looked like a legitimate CRM platform if you only checked the surface. But when I did a subdomain sweep, checking whether api.chaicp.com, admin.chaicp.com, panel.chaicp.com etc. actually existed, something unexpected happened.</p><p>Every single one of them resolved. And every single one pointed to the same place:</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*P6ytXYplxpASv3q997-oaw.png" /></figure><p>agent.juming.com.</p><p>juming.com (聚名) is a Chinese SaaS platform, a <strong>Scam-as-a-Service</strong> engine. Every fake exchange in this network is just a branded skin on top of <a href="http://juming.com/">juming.com</a>&#39;s backend. The operator buys the kit, registers their domain (btczio.com, cionusdt.com, whatever), and <a href="http://juming.com/">juming.com</a> handles everything behind the scenes: fake trade execution, fake balance management, victim data collection, operator admin panel. Multiple scam franchisees, one backend.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*1LF9uFTfg2ERUCt63ih6GA.png" /></figure><p>I also found something that could directly identify the operator’s Google account. Buried in juming.com&#39;s TXT records was a <strong>Google Search Console verification token</strong>:</p><pre>google-site-verification=8uyoo15qyNDwLa-mDSddbiVtovxL1DgPoQVuICowvIc</pre><p>Search that exact string on Google. Whatever domains come back are registered to the same account.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*667oLyjd12MVQDm8iwNSaQ.png" /></figure><p>At this point I took what I had to <a href="https://en.fofa.info/result?qbase64=NDcuNzYuMjE0LjQw"><strong>FOFA</strong></a>, a Chinese internet intelligence platform, similar to Shodan but with significantly better coverage of Alibaba Cloud and Chinese domestic infrastructure. I searched for cname=&quot;agent.juming.com&quot;, looking for any other domains across the internet that CNAME to the same backend.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*tHissN5NZmf7465XOXQtFQ.png" /></figure><p>The results confirmed this wasn’t one operator running one campaign. Multiple deployments, different domain sets, all routing through the same <a href="http://juming.com/">juming.com</a> engine. Different brands, different fake exchange names, one shared backend. This is what a franchise looks like.</p><h3>Step 5, The Second Cluster. 80 Domains.</h3><p>While mapping the first cluster, I ran a reverse-IP lookup on a second IP that kept showing up in associated DNS records. The result stopped me. (<a href="https://viewdns.info/reverseip/?host=34.150.85.92&amp;t=1">ViewDNS.info</a>)</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*FZCQVzllw6LdBnFgWPILWA.png" /></figure><p>34.150.85.92 was hosting <strong>80 domains</strong>. All fake exchanges. All on the same server. The naming patterns were systematic:</p><ul><li><strong>Fake Binance:</strong> binanusdt.com, binalusdt.com, binvvusdt.com, binttusdt.com...</li><li><strong>Fake Bybit:</strong> bybssusdt.com, bybkkusdt.com, byb-n.com...</li><li><strong>Fake BTC exchanges:</strong> btcaac.com, btcaae.com, btcaai.com...</li></ul><p>This wasn’t a single scammer with a single fake site. This was a production-scale typosquatting factory. Someone was running dozens of simultaneous fake exchanges, each one impersonating a real platform, all pointing at the same GCP server.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*wfDTcCu8Er2eQdfURoF_Fw.png" /></figure><p>The WHOIS for the anchor domain, cionusdt.com, was brief but confirming. Registrant Country: <strong>CN</strong>. Updated <strong>three days</strong> before I received that Facebook message. Registered via <a href="http://gname.com/"><strong>Gname.com</strong></a><strong> Pte. Ltd.</strong>, a Singapore-registered, Chinese-operated low-cost registrar with a reputation for minimal abuse enforcement.</p><h3>Step 6, The Second DNS Provider Had Its Own Secret</h3><p>The second cluster used a completely different DNS provider from the first: share-dns.com and share-dns.net. Not julydns.com. A separate operation, different Cloudflare nameserver pair, different account, different infrastructure. Two independent custom scam DNS networks, both running in parallel.</p><p>The first thing I noticed was that share-dns.net has <strong>no A record</strong> for its root domain. Type dig share-dns.net and you get nothing, just a SOA authority response. It exists purely as a nameserver. It doesn&#39;t want to be visited. It doesn&#39;t want to be found.</p><p>Then I checked where their mail server lived:</p><pre>dig mail.share-dns.net</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*5Fz8_yeqO_soDzxrx53lzw.png" /></figure><p>And then the SPF record for share-dns.net:</p><pre>dig share-dns.net TXT</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*OWI8TDS6-oJKEFO7RT7rew.png" /></figure><p>That -all at the end of the SPF record is the interesting part. Most legitimate services use ~all (soft fail) or list multiple IP ranges. This one has exactly one authorized IP and a hard -all, meaning any email that doesn&#39;t come from 35.215.163.95 is to be explicitly rejected. No fallbacks. No third-party mail services. One IP, locked down tight.</p><p>The operator set up a <strong>dedicated, isolated mail server</strong> inside Google Cloud specifically for this DNS provider, completely separate from the fake exchange hosting, separate from the <a href="http://juming.com/">juming.com</a> backend, its own contained piece of infrastructure. Most scam operations route through shared mail services or just use Gmail. This operator built their own.</p><p>That’s not laziness. That’s compartmentalization. Someone who understands that if one piece of the infrastructure gets flagged and taken down, the rest keeps running.</p><p>The numbered nameserver variants made the same point. a1.share-dns.com through a8.share-dns.com are all pre-registered and all resolve to the same Cloudflare anycast IP. Eight nameserver slots, ready to go, waiting to be handed out to eight batches of domains. This DNS provider wasn&#39;t built for one campaign. It was built for scale.</p><h3>Step 7, The Third Cluster. Fresh Deployment.</h3><p>A third GCP IP, 35.247.183.138, Singapore region, was hosting another cluster of 17+ domains, all registered in April 2026. I pointed nmap at port 443 with service detection and two scripts, ssl-cert to read the TLS certificate, and http-headers to capture the full HTTP response headers:</p><pre>nmap -sV --script ssl-cert,http-headers -p 443 35.247.183.138</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*j9eT-Bw-Ky4pGeuzwTgeDA.png" /></figure><p>Two things jumped out immediately.</p><p>First, buried in the HTTP headers:</p><pre>Set-Cookie: PHPSESSID=374d63fc89d4a1cd7f7827421229cf1e; secure; HttpOnly</pre><p>PHPSESSID is a PHP session cookie. That one line told me the backend wasn&#39;t Node.js or Go or anything modern, it was <strong>PHP</strong>. nginx was the front door taking HTTPS connections, but everything behind it was PHP-FPM handling the API. This is classic Chinese exchange kit architecture, nginx as reverse proxy, PHP doing the actual work.</p><p>Then the SSL certificate block:</p><pre>ssl-cert: Subject: commonName=api.coiusdt.com<br>Subject Alternative Name: DNS:api.coiusdt.com<br>Issuer: commonName=R13/organizationName=Let&#39;s Encrypt<br>Not valid before: 2026-04-05T05:10:44<br>Not valid after:  2026-07-04T05:10:43</pre><p>This is the <strong>default vhost certificate</strong>, the cert nginx serves when it receives a request for a hostname it doesn’t recognise. The fact that it says api.coiusdt.com and not bybbn.com or any of the 17 frontend domains means coiusdt.com is the <strong>real identity</strong> of this server. The one the operator actually built the infrastructure around, before layering all the fake exchange brand names on top as frontends.</p><p>Also worth noting: the cert was issued <strong>April 5</strong> and expires <strong>July 4</strong>, a standard 90-day Let’s Encrypt certificate, meaning the operator is running automated cert renewal, almost certainly via certbot or BT-Panel’s built-in Let’s Encrypt integration. This cluster went live on exactly the same date the SOA serial timestamps pointed to.</p><p>One more thing in those headers that’s easy to miss:</p><pre>Access-Control-Allow-Headers: ... userid</pre><p>That userid field is a <strong>non-standard custom header</strong>, not part of any HTTP spec. The PHP API expects it alongside the session token to identify which user is making a request. It&#39;s a fingerprint unique to this exchange kit. Any server on the internet returning this header in a CORS preflight response is almost certainly running the same platform.</p><p>I took that header to <a href="https://app.netlas.io/responses/?q=http.headers%3Auserid"><strong>Netlas</strong></a>, which indexes HTTP response data across its internet-wide scans. Searching for servers returning this specific userid header combination alongside PHPSESSID would surface any other global deployments of this same PHP kit, regardless of what domain name or brand they were hiding behind.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*RQB9iRNiE0QYgCFV2Iojow.png" /></figure><p>Then I ran openssl with the right SNI flag, and the TLS certificate handed me seven new domains for free.</p><h3>Step 8, One openssl Command. Seven New Targets.</h3><pre>echo | openssl s_client -connect 35.247.183.138:443 \<br>  -servername bybbn.com 2&gt;/dev/null \<br>  | openssl x509 -noout -text | grep -A5 &quot;Subject Alternative&quot;</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*7QVaCraLXNmsvRXsW1o3eg.png" /></figure><p>Let’s Encrypt issues multi-domain certificates. When an operator registers a batch of domains in a single certbot call, all those domains get bundled into one cert’s Subject Alternative Name list. Pull the cert, read the SAN, you get the entire batch.</p><p>From one command, seven previously unknown domains: bincusdt.com, binnusdt.com, bybkh.com, fs-wbwx.com, gis-tds.com, pp-aybz.com, y-bwsr.com. All on the same server. All part of the same operator&#39;s Wave 1 deployment on April 5.</p><p>The natural follow-up question is: what did they deploy <em>before</em> this? Certificate Transparency logs are public, every cert ever issued by Let’s Encrypt or any major CA gets written to an append-only public log permanently. <a href="https://crt.sh/?q=bybbn.com"><strong>crt.sh</strong></a> indexes all of it.</p><p>I searched %.coiusdt.com and %.cionusdt.com.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*f9KGAmuYxwanrs2AZl93kA.png" /></figure><p>Every subdomain the operator had ever requested a cert for came back, including ones that no longer existed in DNS. Retired infrastructure. Previous deployments. Subdomains that had been spun up and taken down before this campaign started. The cert logs remember everything even when the DNS records are gone.</p><p>I also took the SHA-1 fingerprint from the nmap output (942c15a039072c37b1b578dadb7d2ba876c292b1) to <a href="https://platform.censys.io/search?q=%22942c15a039072c37b1b578dadb7d2ba876c292b1%22"><strong>Censys</strong></a> and searched for other hosts presenting certificates from the same issuer chain and key pattern.</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*bHKG6TxC7haDSZapJ_O8ZQ.png" /></figure><p>If the operator had migrated servers recently, the old host might still be serving the same cert.</p><h3>Step 9, Inside the Fake Exchange</h3><p>I did a simple curl request, using the --resolve flag to point bybbn.com at the known IP without touching real DNS, and read the HTML source of the actual page.</p><pre>curl -sk --resolve &#39;bybbn.com:443:35.247.183.138&#39; https://bybbn.com/</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*RoOFR8tmwxUT2a3vPpck1g.png" /></figure><p>The site was a Vue.js single-page application, a professionally built web app serving a fake cryptocurrency exchange interface. Clean UI, real-time price tickers, trading charts. Indistinguishable from the real thing if you don’t know what to look for.</p><p>But one line in the HTML gave everything away:</p><pre>&lt;script&gt;<br>  window.wss = &#39;wss://chat.cionusdt.com/&#39;<br>&lt;/script&gt;</pre><p>Every single fake exchange domain in Cluster C, all 25+ of them, has this exact line hardcoded into the HTML. They all connect to the <strong>same WebSocket server</strong>: chat.cionusdt.com on the second cluster (34.150.85.92).</p><p>This is the scammer’s real-time control channel. One operator, sitting at one chat panel, can simultaneously manage hundreds of victims across dozens of fake exchange frontends. When a victim types into the “customer support” chat, that message goes to chat.cionusdt.com. The scammer responds as &quot;support agent,&quot; denies the withdrawal, demands the next VIP payment.</p><p>One server. Hundreds of victims. One person orchestrating it all.</p><h3>Step 10, The Source Code Confessed Everything</h3><p>I had captured a complete mirror of <a href="http://btczio.com/">btczio.com</a>, 35 files including the full JavaScript source. Inside were two complete bundles: one from <strong>April 14</strong> and one from <strong>April 24</strong>. Ten days apart.</p><p>I ran a targeted extraction on both files looking for the API backend URL. The difference was immediate:</p><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*_mJis2veaRAIHjvRbVBaNA.png" /></figure><p>The operator had <strong>migrated their entire backend</strong> ten days before the attack reached this inbox. The old backend, api.nixmss.com, resolves to 35.197.129.159: the same IP as <a href="http://btczio.com/">btczio.com</a> itself. A previously unknown domain, predating this campaign. And the new one, api.coiusdt.com, is what the live site was running on when I found it.</p><p>nixmss.com is a new IOC that nobody was tracking.</p><p>Then I found the name.</p><h3>Step 11, They Named It</h3><p>Deep in the main JavaScript bundle, a single constant. The platform’s internal configuration:</p><pre>const l0 = {title: &quot;Redwood Trus&quot;, lang: &quot;zh&quot;}</pre><figure><img loading="lazy" referrerpolicy="no-referrer" alt="" src="https://cdn-images-1.medium.com/max/1024/1*TbSjAAaFLLtZVBp94yJu4w.png" /></figure><p><strong>Redwood Trus.</strong> That’s what they call it. The fake exchange brand name, hardcoded in source. The truncation, “Trus” instead of “Trust”, isn’t a typo. It’s deliberate. Exact-match brand protection searches, fraud detection systems, and Google Safe Browsing all use string matching. “Redwood Trust” would hit filters. “Redwood Trus” slides past them.</p><p>The lang:&quot;zh&quot; next to it, combined with Chinese-language developer console logs throughout the code (连接正常, 开启心跳, 出现错误, &quot;Connection normal&quot;, &quot;Heartbeat started&quot;, &quot;Error occurred&quot;), confirms what the infrastructure already suggested: built by Chinese developers, for Chinese-operated fraud.</p><h3>What We Started With.
